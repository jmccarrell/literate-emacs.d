* Purpose

This document is the Phase 0 deliverable of the modernization plan: a
map from Jeff's current ~jeff-emacs-config.org~ (last seriously
refactored around 2020) to the state of the Emacs ecosystem in 2026,
plus a prioritized backlog of sub-goals.

It is a reference doc, not part of the tangled config. Keep it in the
~main/~ worktree and update it as sub-goals land.

* TLDR — what actually changed 2020 → 2026

Four things matter most, in rough order of impact on Jeff's config:

1. *The completion stack flipped.* The 2020 default of
   ivy/counsel/swiper has been displaced in new configs by the
   vertico + consult + marginalia + embark + orderless ecosystem
   (authored by Daniel Mendler). Ivy is still maintained (Basil
   Contovounesios now leads; 0.15.x released 2025) — but the
   center of gravity, including /abo-abo's own/ config, moved
   toward eglot + orderless-style filtering. Not a crisis; a
   weather report.

2. *LSP, tree-sitter, use-package, and which-key are now built in.*
   Emacs 29 (July 2023) bundled use-package and eglot and added
   tree-sitter (~treesit.el~). Emacs 30 (Feb 2025) added
   which-key as built-in and ~:vc~ to use-package (install
   directly from a git repo, no package manager gymnastics). This
   collapses a lot of third-party bootstrap code.

3. *Python dev changed out from under elpy.* Jeff already noted
   elpy was deprecated. The 2026 answer is eglot + a modern LSP:
   basedpyright (a faster/community fork of pyright) for types,
   ruff-lsp for lint/format, optionally ty (new, fast Python
   type-checker from the ruff team). ~python-ts-mode~ replaces
   ~python-mode~ where tree-sitter is available. uv is
   increasingly replacing pyenv+poetry.

4. *AI showed up and is now a real dimension of the config.* This
   didn't exist in 2020. In 2026: gptel is the de-facto general
   LLM client (Claude, GPT, Gemini, local via ollama, MCP
   tool-use); copilot.el or claudemacs for inline completion;
   aidermacs for autonomous refactoring; local models via
   llama-swap / ollama / LiteLLM. jwiegley's config (the most
   sophisticated reference in the set) is now /dominated/ by
   AI-related modules.

Secondary shifts that matter: flycheck → flymake (because eglot
ships with flymake); projectile → ~project.el~ (built-in is now
good enough for most); company → corfu+cape; ag → consult-ripgrep;
zenburn is fine but the modern accessibility-first default is
modus-themes (also built in since 28.1); expand-region has a
leaner successor in ~expreg~.

* Prioritized backlog

** Shipped

The rough order the sub-goals landed in:

- *Phase 0 landscape survey* (this document).
- *Markdown rendering pilot* — =gfm-mode= with native code-block
  fontification, heading scaling, =edit-indirect= for =C-c '= on
  fenced blocks.
- *Cheat sheet — markdown mode* — documented bindings as its own
  section in =emacs-cheat-sheet.org=.
- *Completion stack migration* — ivy/counsel/swiper/counsel-
  projectile/ivy-hydra/ag replaced with vertico + consult +
  marginalia + embark + embark-consult + orderless +
  consult-ripgrep. Savehist and vertico-directory wired.
  Projectile kept with =default= completion-system. =C-.= →
  embark-act; =C-;= → avy. Cheat sheet updated.
- *Emacs-30 cleanups* — dropped use-package bootstrap; built-in
  =which-key= and =editorconfig= (=:ensure nil=); fixed elpa-mirror
  =cached-package-dir= interpolation bug; swapped
  =defalias 'yes-or-no-p= for =(setq use-short-answers t)=.
- *Python + LSP + tree-sitter* — =python-mode=/elpy/flycheck →
  built-in =python.el= (with =python-ts-mode= remap) + eglot +
  ty + ruff + flymake + flymake-ruff. Tree-sitter grammar
  bootstrap helper =jwm/ensure-treesit-grammars= plus source-alist
  for python/json/yaml/toml. Mode remaps for json/yaml/toml.
  Cheat sheet updated (=C-c '= for edit-indirect + a new Code
  blocks subsection).
- *Extend LSP / tree-sitter coverage* — grammars + remaps for
  go/dockerfile/javascript/just. Go gets eglot + gopls (default) +
  =goimports-format-on-save-mode= replacing =gofmt-before-save=.
  Dockerfile gets eglot + =docker-langserver= (delegates to
  hadolint for lint). Justfile gets =just-ts-mode= alongside
  =just-mode=. JavaScript gets ts-mode remap, no LSP.
- *zsh → bash shell config migration* — replaced the multi-level
  zsh-specific =-conorcs= / =-cf= optimization with a brief
  =** Terminal and Shell Settings= section. Bash's
  non-interactive default is fast enough; no
  =shell-command-switch= override needed. =jwm/shell-is-zsh-p=
  removed.
- *Drop dash / s / f libraries* — investigation found zero call
  sites in the config; the =** Howard's Recommended Emacs Lisp
  Libs= section was vestigial. Dropped the three =use-package=
  forms (transitive deps via magit etc. still keep the libs
  installed in =~/.emacs.d/elpa/=).
- *Investigation: batch-mode tangling* — confirmed
  =emacs --batch -l org --eval '(org-babel-tangle-file ...)'=
  produces byte-identical output to interactive
  =M-x org-babel-tangle= (empty diff against
  =main/init.el=, ~480ms wall-clock cold). Recommended a
  =justfile= recipe; implementation shipped below.
- *Add justfile with tangle and verify-tangle recipes* — added
  =literate-emacs.d/main/justfile= with =tangle= (regenerates
  =init.el=), =verify-tangle= (tangles + loads in batch =-Q= to
  catch load errors), and a private =@_= recipe that prints the
  list. Workspace =CLAUDE.md= updated to document the commands.
- *Drop hydra use-package declaration* — investigation found
  zero call sites in the config (no =defhydra=, no =(hydra-...)=
  calls, no =/body=). The =** Hydra= section was vestigial.
  For any future menu-style commands, prefer =transient=
  (built-in Emacs 28+; what magit uses).
- *Add info-tooling* — two new justfile recipes for grounding
  investigations in Jeff's specific Emacs install state.
  =info-dir-update= regenerates a tracked =info-dir.txt= snapshot
  of all visible Info manuals. =info-node "(...)"= dumps a
  specific node to =/tmp/info-node.txt= for attaching to a
  session. Workspace =CLAUDE.md= documents the standard workflow:
  investigations start by reading =info-dir.txt=.

*** Terraform LSP + format-on-save

    Jeff has =terraform-mode= (MELPA) installed but no language
    server or formatter wired up. Mirror the Python sub-goal
    pattern in the existing =** Eglot= and =** Flymake= sections:

    - Add =terraform-ls= (HashiCorp's official LSP) to
      =eglot-server-programs= for =terraform-mode=.
    - Add =eglot-ensure= hook to =terraform-mode=.
    - Define a =reformatter= for =terraform fmt= and enable a
      =terraform-format-on-save-mode= hook (mirror of
      =ruff-format-on-save-mode=).
    - Flymake diagnostics flow through eglot automatically; no
      extra wiring needed.

    No tree-sitter work: Emacs 30 has no built-in
    =terraform-ts-mode=, and the HCL/Terraform grammar story is
    community-only. =terraform-mode= still handles the buffer.

    *Alternative:* =tofu-ls= for OpenTofu (HashiCorp's license
    change drove some users to OpenTofu; Purcell switched
    recently). Pick based on which CLI you actually use.

    *Preconditions:* =brew install terraform-ls= (or =tofu-ls=).
    The =terraform= binary already needs to be on =PATH= for
    =terraform fmt= to work.

    *Effort:* Low — 30-60 minutes. Simpler than Python (one LSP,
    no type-checker + linter split).

    *Reference:* Purcell's =Define a reformatter for terraform=
    and =Add tofu-ls as an eglot LSP for terraform= commits
    (~2025-2026).

*** Investigate uv for Python

    =uv= is astral's fast Python package/project manager (same team
    as ruff and ty). It subsumes pip / pipx / poetry / pyenv /
    virtualenv. Some of Jeff's current python tooling path — pyenv
    for interpreter management, pipx for =ty= and =ruff= — is
    orthogonal to what uv replaces; some of it may be redundant.

    Scope of the investigation (research, not implementation):

    - What does Purcell actually do? His =init-python.el= has a
      =uv.lock → toml-mode= entry (a small signal that he opens
      uv projects). Is there more downstream?
    - What does =uv-mode= (MELPA) provide? Useful or overkill?
    - Does =eglot-python-preset= or =emacs-pet= already know about
      uv venvs, or do we need to wire it up manually?
    - Should we install =ty= / =ruff= / =basedpyright= via
      =uv tool install= instead of =pipx install=, for
      consistency?
    - Does uv's project venv (=.venv= in project root) get
      auto-picked-up by eglot + ty when opening a file?

    *Output:* a short write-up plus a recommendation — either "no
    action needed, uv Just Works with the current setup" or
    "here's a proposed sub-goal to wire in uv support." The
    investigation itself is the deliverable; any implementation
    would be a separate follow-on sub-goal.

    *Effort:* Low — an hour of reading Purcell, the uv docs, and
    testing on a sample project.

    *Reference:* Purcell's current =init-python.el=;
    https://docs.astral.sh/uv/; =uv-mode= on MELPA;
    =emacs-pet= on MELPA.

*** Investigate eat (terminal emulator, vterm alternative)

    =eat= ("Emulate a Terminal") is a pure-elisp terminal emulator
    package available on GNU ELPA. Purcell adopted it recently per
    his commit log (=Eat terminal tweaks=, =Don't keep adding the
    same keybinding to eat-semi-char-non-bound-keys=). The case
    for it over =vterm= is portability — vterm needs libvterm + a
    C module build step; eat doesn't.

    Scope of the investigation (research, not implementation):

    - What does Purcell configure for eat?
    - How does eat compare to =ansi-term= / =term= / =shell= for
      Jeff's typical use (running =git=, =pytest=, etc.)?
    - Is Jeff currently using any in-Emacs terminal at all? The
      current config has no =vterm=, =eat=, or =ansi-term=
      customization — he may be entirely shell-out-to-iTerm. If
      so, is there a reason to add a terminal to Emacs now?
    - Performance claims vs measured reality.

    *Output:* a short write-up plus recommendation. If yes, a
    sub-goal proposal for wiring eat into the config.

    *Effort:* Low — 30-60 minutes to read Purcell's commits, the
    eat README, and try it briefly.

    *Reference:* Purcell's eat-related commits in =init.el=;
    https://codeberg.org/akib/emacs-eat.

*** Investigate =verb= (org-babel HTTP client)

    =verb= is a newer org-mode HTTP client with per-request
    variables, environment files, and richer response handling
    than =ob-restclient= (which Jeff currently uses). Evaluate
    whether the added features justify a switch.

    *Effort:* Low — 30 min of docs reading + sample comparison.
    *Reference:* https://github.com/federicotdn/verb

*** Investigate =org-modern=

    =org-modern= (minad, same author as vertico/consult) makes
    org buffers look dramatically nicer (cleaner bullets, boxed
    headings, syntax-highlighted keywords) without changing
    semantics. Widely adopted in 2024+ configs.

    Check interaction with =zenburn= — some themes compose well
    with org-modern, some clash.

    *Effort:* Low — 15 minutes to enable and eyeball.
    *Reference:* https://github.com/minad/org-modern

*** Investigate magit ecosystem additions

    Three independent, orthogonal packages — evaluate each
    separately:

    - =magit-todos= — surfaces TODO/FIXME/HACK in magit-status.
    - =magit-delta= — routes magit diffs through =delta= (rust)
      for syntax-highlighted output.
    - =difftastic= — tree-sitter-based structural diff; can
      replace =git diff= or be invoked from magit.

    *Effort:* Low — 15-30 min per package.

*** Investigate =dirvish= (modern dired front-end)

    =dirvish= wraps dired with a sidebar, preview pane, file-
    info header, and IDE-like browsing. Some users love it,
    some find it overkill for dired's "buffer of text"
    minimalism.

    Precondition question: does Jeff use dired frequently
    enough to justify the added surface?

    *Effort:* Low — try it for a day, see if it sticks.
    *Reference:* https://github.com/alexluigit/dirvish

*** Investigate magit's git worktree support

    Jeff has been using git worktrees heavily in this project (and
    in other personal / professional repos). Magit ships some
    worktree support — accessible via the magit-status dispatch
    transient — but the exact capabilities and how they compare
    to the shell =git worktree= commands haven't been mapped.

    Scope of the investigation:

    - What does magit's worktree UI cover? Add, remove, switch,
      list, branch-from-worktree?
    - How does it compare to running =git worktree= by hand?
      Better discoverability, but does it cover the flags
      typically used (=-b <branch>=, =--no-checkout=, etc.)?
    - Does it integrate with project.el or projectile so jumping
      to "the markdown-rendering worktree" is one keystroke
      from anywhere?
    - Could it replace parts of the manual command block in the
      workspace =CLAUDE.md= for setting up sub-goal worktrees?
      (Note: our setup also creates =info/=, the =TASK.md=
      exclude, and the =init.el= symlink — those are
      project-specific and magit won't do them automatically.)
    - Are there third-party helpers that fill gaps (e.g. a
      function wrapping magit + our boilerplate)?

    *Output:* a write-up + recommendation. Likely outcomes:
    - "Use magit's worktree commands for the git plumbing;
      everything else stays manual."
    - "Wrap our setup boilerplate in a =jwm/start-sub-goal-worktree=
      function that invokes magit-worktree-checkout-new and then
      handles =info/exclude=, =TASK.md=, and the symlink."
    - "Magit's worktree UI doesn't add much; stay with shell."

    *Effort:* Low — 30-45 min of poking around in =magit-status=,
    reading the manual, and pondering integration.

    *Reference:* magit user manual,
    https://magit.vc/manual/magit/Worktree.html

*** Investigate modern undo patterns

    Jeff uses the built-in =undo= via =C-x u=. It works — and
    it's actually a tree, not linear — but the UX is notoriously
    confusing: undo-then-type creates a branch that's hard to
    recover. Three 2026 options to compare:

    - =vundo= (Casouri, 2022+) — visualizes the built-in undo
      tree without replacing the underlying system. Lightweight,
      well-regarded, no format lock-in.
    - =undo-fu= + =undo-fu-session= — minimalist linear
      undo/redo semantics (closer to Ctrl-Z in other editors);
      session package persists history across restarts.
    - =undo-tree= (longstanding) — visualizes and navigates
      branches; older than vundo; some known concerns about its
      on-disk storage format at scale.

    Scope of the investigation:
    - What do Purcell / bbatsov / jwiegley use in their current
      configs? Good chance of a clear community signal.
    - Demo each on a sample buffer to see which feels best
      for Jeff's editing style.
    - Keybinding question: stay on =C-x u=, or add Ctrl-Z /
      Ctrl-Shift-Z bindings that match other editors? (Note:
      =C-z= is =suspend-frame= by default — frequently rebound.)

    *Output:* a write-up + recommendation. If a clear winner, a
    follow-on sub-goal to wire it in.

    *Effort:* Low — 30-60 min of reading + testing on a scratch
    buffer.

    *References:* https://github.com/casouri/vundo;
    https://codeberg.org/ideasman42/emacs-undo-fu;
    https://www.dr-qubit.org/undo-tree.html

* Known open questions / things to verify

These are items I want to double-check before committing to them
in a sub-goal proposal:

1. Which Emacs version Jeff is actually running. The config
   comment "As of emacs 29, use-package is built in" suggests 29,
   but the config still has the ~package-initialize~ pattern
   and doesn't use ~:vc~. Emacs 30 (Feb 2025) is current stable;
   Emacs 31 is in development. This matters for the "built-in
   which-key", ~:vc~, and built-in editorconfig calls. Run
   ~M-x emacs-version~.

2. ~exec-path-from-shell~. Jeff decided against it in 2020 on
   the grounds that he always invokes Emacs from iTerm. That
   decision is still fine, but worth revisiting if Emacs starts
   auto-launching outside a shell (e.g. via macOS Launchpad)
   because of something downstream (e.g. eglot needing
   ~PYENV_ROOT~).

3. macOS Emacs build. Jeff uses ~brew install emacs-app~. The
   2026 field contains at least emacs-app, emacs-plus,
   emacs-plus-app (new 2026 cask), and mitsuharu's Mac port.
   If Jeff is still on an older emacs-app without native
   compilation, an upgrade here is free performance.

4. ~elpa-mirror~ offline setup. This is unique to Jeff. Any
   migration that adds new packages needs to be tested through
   the mirror build step. Not a blocker, but keep it in mind.

* How to use this document

- Treat the /Section-by-section mapping/ as Jeff's backlog.
- Pick a sub-goal. Spin up a feature worktree
  (~literate-emacs.d/<feature>/~). Drop a ~TASK.md~ in it
  describing the goal and approach, per the workspace
  convention.
- Follow the five-step loop: research → evaluate → propose →
  implement → verify.
- Come back here after each sub-goal lands; revise the table.

* Purpose

This document is the Phase 0 deliverable of the modernization plan: a
map from Jeff's current ~jeff-emacs-config.org~ (last seriously
refactored around 2020) to the state of the Emacs ecosystem in 2026,
plus a prioritized backlog of sub-goals.

It is a reference doc, not part of the tangled config. Keep it in the
~main/~ worktree and update it as sub-goals land.

* TLDR — what actually changed 2020 → 2026

Four things matter most, in rough order of impact on Jeff's config:

1. *The completion stack flipped.* The 2020 default of
   ivy/counsel/swiper has been displaced in new configs by the
   vertico + consult + marginalia + embark + orderless ecosystem
   (authored by Daniel Mendler). Ivy is still maintained (Basil
   Contovounesios now leads; 0.15.x released 2025) — but the
   center of gravity, including /abo-abo's own/ config, moved
   toward eglot + orderless-style filtering. Not a crisis; a
   weather report.

2. *LSP, tree-sitter, use-package, and which-key are now built in.*
   Emacs 29 (July 2023) bundled use-package and eglot and added
   tree-sitter (~treesit.el~). Emacs 30 (Feb 2025) added
   which-key as built-in and ~:vc~ to use-package (install
   directly from a git repo, no package manager gymnastics). This
   collapses a lot of third-party bootstrap code.

3. *Python dev changed out from under elpy.* Jeff already noted
   elpy was deprecated. The 2026 answer is eglot + a modern LSP:
   basedpyright (a faster/community fork of pyright) for types,
   ruff-lsp for lint/format, optionally ty (new, fast Python
   type-checker from the ruff team). ~python-ts-mode~ replaces
   ~python-mode~ where tree-sitter is available. uv is
   increasingly replacing pyenv+poetry.

4. *AI showed up and is now a real dimension of the config.* This
   didn't exist in 2020. In 2026: gptel is the de-facto general
   LLM client (Claude, GPT, Gemini, local via ollama, MCP
   tool-use); copilot.el or claudemacs for inline completion;
   aidermacs for autonomous refactoring; local models via
   llama-swap / ollama / LiteLLM. jwiegley's config (the most
   sophisticated reference in the set) is now /dominated/ by
   AI-related modules.

Secondary shifts that matter: flycheck → flymake (because eglot
ships with flymake); projectile → ~project.el~ (built-in is now
good enough for most); company → corfu+cape; ag → consult-ripgrep;
zenburn is fine but the modern accessibility-first default is
modus-themes (also built in since 28.1); expand-region has a
leaner successor in ~expreg~.

* What changed in Jeff's reference repos

I re-fetched all 14 reference repos and compared local HEAD to
current origin. Here is the signal-sorted result.

** High signal (actively evolved)

*** jwiegley-dotemacs — 2024-11 → 2026-04, 396 commits

   *The most sophisticated reference.* His recent commits are
   /overwhelmingly/ AI-related: ~gptel-presets.el~,
   ~gptel-ext.el~, ~gptel-backends.el~, ~gptel-agent~ submodule,
   ~gptel-rag.el~, ~gptel-prompts~, ~hf.el~ (local Hugging Face
   models, LiteLLM config generation, llama-swap orchestration),
   ~agent-shell~, ~mcp-server-lib~, ~macher~, ~magit-ai~. He
   runs multiple local LLMs (Qwen3 series, Devstral) behind
   LiteLLM and switches between them and cloud (Claude Opus/
   Sonnet 4-6) via gptel presets.

   Non-AI touches: relint, magit ANSI coloring, org-roam-sync,
   capture-link-to-entry, org-inlinetask tweaks.

   *Takeaway:* if you want to see the ambitious ceiling of emacs
   + AI, this is it. Also: jwiegley didn't rip out fundamentals;
   the 2020 completion stack and organization are broadly
   preserved. AI is /additive/.

*** steve-purcell-dotemacs — 2018-06 → 2026-04, 576 commits

   *The best migration reference for Jeff.* Purcell's
   chronological diff is essentially the exact path from where
   Jeff is to modern:
   - ivy → selectrum → vertico (init-minibuffer.el is now the
     active file; init-ivy.el is legacy)
   - added consult + embark; ~consult-ripgrep-at-point~ wrapper
   - flycheck → flymake (has init-flymake.el)
   - company → corfu (with careful Emacs version guards)
   - added tree-sitter (init-treesitter.el) then /removed/
     treesit-auto once Emacs 30 covered his needs natively
   - eglot everywhere, with LSPs: ty and basedpyright for
     Python; nixd for Nix; tofu-ls for Terraform; ocaml-ts-mode
     with LSP; clojure-ts-mode + cider
   - ~ruff-fix-on-save-mode~ for Python
   - Removed ~ag~ (dropped in favor of consult-ripgrep)
   - Added pulsar (visual feedback on cursor jumps), eat
     (modern terminal, alternative to vterm)
   - Added ns-auto-titlebar for macOS
   - Uses pyproject.toml and uv.lock as project root markers

   *Takeaway:* walk ~git log~ on this repo when planning any
   sub-goal. He has /already done/ the migration you are
   contemplating, with clean small commits.

*** bbatsov-dotemacs — 2021-07 → 2026-04, 95 commits

   Parallel modernization to Purcell, but with different
   opinions:
   - Dropped use-package bootstrap (built-in since 29)
   - Moved which-key to "built-in packages" section (built-in
     since 30)
   - Dropped ~ag~ → consult-ripgrep
   - Dropped elixir-mode, rust-mode, yaml-mode in favor of
     built-in ts-modes
   - Dropped treesit-auto (redundant on Emacs 30+)
   - Switched from expand-region to ~expreg~
   - Added copilot
   - Eglot auto-shutdown when no longer needed
   - Switched theme to tokyo-night / tokyo-storm
   - Enabled ~repeat-mode~ with a custom repeat-map for expreg
   - Added ~minibuffer-regexp-mode~, ~visual-wrap-prefix-mode~
   - ~read-extended-command-predicate~ to filter ~M-x~
   - File-notification-based auto-revert

   *Takeaway:* bbatsov is a great source of small,
   self-contained quality-of-life upgrades. Many of these are
   one-line adoptions.

*** abo-abo-dotemacs — 2021-10 → 2025-12, 227 commits

   *Significant:* abo-abo /is the author/ of ivy/counsel/swiper/
   avy. Yet his own recent commits include:
   - ~modes/ora-python.el: Fix eglot completion delay~ (he uses
     eglot for Python!)
   - ~init.el: Don't load ora-lsp~ (dropped lsp-mode)
   - ~init.el: Load ora-flycheck~, ~ora-vterm~ (still uses
     flycheck, still uses vterm)
   - Still uses ~ora-ivy.el~, ~ora-company.el~ — so he hasn't
     migrated off his own stack, but the LSP world has moved on
     around him.

   *Takeaway:* the ivy author uses eglot. That's the story.
   Ivy is not dead, but depending on it for LSP UX is no longer
   where the maintenance is.

*** munen-emacs.d — 2019-07 → 2026-04, 336 commits

   Another AI-heavy evolution, but a different flavor than
   jwiegley: munen has built an extensive set of /gptel tools/
   (custom functions exposed to the LLM as tools):
   ~read_file~, ~edit_file~, ~edit_file_non_interactive~,
   ~apply_diff~, ~replace_buffer~, ~list_directory~,
   ~file_lint_with_flycheck~, ~run_clojure_in_repl~. Uses
   ~agent-shell~ with ~opencode~ as default agent; uses the
   ~mcp~ package. Packages managed via Guix.

   *Takeaway:* if the goal is "make emacs an agentic coding
   environment driven by an LLM I trust," munen's pattern (gptel
   + custom tools + MCP) is one viable shape. Less polished than
   jwiegley but more compact.

*** sacha-chua-dotemacs — 2019-09 → 2026-04, 262 commits (gh-pages branch)

   Organic evolution rather than a clean migration; Sacha's use
   of Emacs is heavily blogging + life-admin + transcript work
   (whisperx for audio, ox-11ty for static site). Relevant
   signals: she adopted marginalia, ~minibuffer-regexp-mode~,
   ~goto-chg~, ~substitute~. Explicit "more tips from bbatsov"
   commits show cross-pollination. Still on vertico/marginalia
   stack; not on the AI bandwagon as deeply as jwiegley/munen.

* Prioritized backlog

** Shipped

The rough order the sub-goals landed in:

- *Phase 0 landscape survey* (this document).
- *Markdown rendering pilot* — =gfm-mode= with native code-block
  fontification, heading scaling, =edit-indirect= for =C-c '= on
  fenced blocks.

  section in =emacs-cheat-sheet.org=.

  projectile/ivy-hydra/ag replaced with vertico + consult +
  marginalia + embark + embark-consult + orderless +

  embark-act; =C-;= → avy. Cheat sheet updated.

  =which-key= and =editorconfig= (=:ensure nil=); fixed elpa-mirror
  =cached-package-dir= interpolation bug; swapped
  =defalias 'yes-or-no-p= for =(setq use-short-answers t)=.

  built-in =python.el= (with =python-ts-mode= remap) + eglot +
  ty + ruff + flymake + flymake-ruff. Tree-sitter grammar
  bootstrap helper =jwm/ensure-treesit-grammars= plus source-alist
  for python/json/yaml/toml. Mode remaps for json/yaml/toml.

  blocks subsection).

** Pending

*** AI / LLM integration

    New section in the config. Start narrow: gptel with a Claude
    backend for general chat. Optionally add one of: copilot.el
    for inline completion, aidermacs for agentic refactoring,
    local-via-ollama for offline use. Keep the first landing
    small.

    *Effort:* Low for basic chat (30 min). Medium if you want
    custom tools / MCP / local models — that's a rabbit hole.
    jwiegley's and munen's configs are the two extremes to
    calibrate against.

    *Caveat:* this area moves monthly. Whatever we pick in week
    1 may have a better alternative in week 12. The plan should
    bias toward reversibility.

*** zsh → bash shell config migration

    The =* Variables → ** Terminal and Shell (zsh) Settings=
    section is entirely zsh-specific (the =-conorcs= discussion,
    =jwm/shell-is-zsh-p=, the =shell-command-switch= override).
    Jeff no longer uses zsh — the interactive shell is now bash.

    Bash's behavior is simpler: non-interactive shells
    (=bash -c=) don't read =.bashrc= or =.bash_profile= by default,
    so the whole "optimize startup" dance the zsh section
    describes is unnecessary for bash. The migration is therefore
    mostly /deletion/:

    - Remove =jwm/shell-is-zsh-p= and the =shell-command-switch=
      override (bash's default =-c= is already fast).
    - Rewrite or remove the prose about zsh invocation flags —
      mostly historical at this point.
    - Rename section to something shorter, e.g.
      =** Terminal and Shell Settings=.

    *Effort:* Low — 30-60 minutes including verification.

*** Small cleanup pass (accumulated drift from earlier sub-goals)

    Each item is too small to deserve its own sub-goal; together
    they make a reasonable 15-30 minute pass.

    - Drop the MELPA =json-mode= and =yaml-mode= packages — now
      vestigial since the tree-sitter remaps do the work. Replace
      their =:mode= directives with bare =auto-mode-alist= entries
      pointing directly at =json-ts-mode= / =yaml-ts-mode=.
    - Remove the =(markdown-command "pandoc ...")= line in the
      Markdown section — preview isn't configured and the line is
      latent-error material.
    - Swap =expand-region= for =expreg= (bbatsov's modern default).
      Update cheat sheet.
    - Check whether =hydra= is still used anywhere post-ivy
      migration; drop the =use-package hydra= form if no consumers
      remain.
    - Switch =js2-mode= to =js-ts-mode= (install javascript
      grammar; add to =jwm/required-treesit-grammars=;
      =major-mode-remap-alist= entry).

    *Effort:* Low. One worktree, one or two commits.

* Known open questions / things to verify

These are items I want to double-check before committing to them
in a sub-goal proposal:

1. Which Emacs version Jeff is actually running. The config
   comment "As of emacs 29, use-package is built in" suggests 29,
   but the config still has the ~package-initialize~ pattern
   and doesn't use ~:vc~. Emacs 30 (Feb 2025) is current stable;
   Emacs 31 is in development. This matters for the "built-in
   which-key", ~:vc~, and built-in editorconfig calls. Run
   ~M-x emacs-version~.

2. ~exec-path-from-shell~. Jeff decided against it in 2020 on
   the grounds that he always invokes Emacs from iTerm. That
   decision is still fine, but worth revisiting if Emacs starts
   auto-launching outside a shell (e.g. via macOS Launchpad)
   because of something downstream (e.g. eglot needing
   ~PYENV_ROOT~).

3. macOS Emacs build. Jeff uses ~brew install emacs-app~. The
   2026 field contains at least emacs-app, emacs-plus,
   emacs-plus-app (new 2026 cask), and mitsuharu's Mac port.
   If Jeff is still on an older emacs-app without native
   compilation, an upgrade here is free performance.

4. ~elpa-mirror~ offline setup. This is unique to Jeff. Any
   migration that adds new packages needs to be tested through
   the mirror build step. Not a blocker, but keep it in mind.

* How to use this document

- Treat the /Section-by-section mapping/ as Jeff's backlog.
- Pick a sub-goal. Spin up a feature worktree
  (~literate-emacs.d/<feature>/~). Drop a ~TASK.md~ in it
  describing the goal and approach, per the workspace
  convention.
- Follow the five-step loop: research → evaluate → propose →
  implement → verify.
- Come back here after each sub-goal lands; revise the table.
